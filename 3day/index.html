Session 3.1

Classic Path 개념 및 활용

단계적 마이그레이션을 위한 연결 통로

Classic Path란?

Classic 환경과 VPC 환경을 사설 통신으로 연결하는 서비스입니다.

Classic 서버

Private 통신
Classic Path

VPC 서버

활용 사례: Classic의 DB를 그대로 사용하면서 VPC 환경에 신규 웹 서버를 구성하는 경우

Classic Path 설정 절차

  1. 1

    Classic Path 신청

    콘솔에서 연결할 Classic Zone과 VPC를 지정하여 서비스를 신청합니다.

  2. 2

    VPC 라우팅 테이블 설정

    Classic 서버 대역으로 향하는 트래픽(Destination)의 Target을 Classic Path로 지정합니다.

  3. 3

    양방향 보안 정책 설정

    Classic의 ACG와 VPC의 ACG/NACL에서 상대방 대역과의 통신을 허용합니다.

Session 3.2

DMS 활용 실습

Database Migration Service

Database Migration Service (DMS) 란?

최소한의 중단

초기 데이터 복제 후 변경분(CDC)을 지속적으로 동기화하여 서비스 중단을 최소화합니다.

이기종 마이그레이션

MySQL to MySQL 뿐만 아니라 Oracle to MySQL 등 다른 종류의 DB 간 이전도 지원합니다.

간편한 관리

콘솔에서 몇 번의 클릭만으로 마이그레이션 작업을 생성하고 모니터링할 수 있습니다.

DMS 마이그레이션 절차

1. Source DB

Classic 환경의 원본 DB

2. Endpoint 설정

Source/Target DB 접속 정보 등록

3. Migration Task 생성

이전 유형(Full+CDC) 선택 및 실행

4. Target DB

VPC 환경의 대상 DB

실습: Classic MySQL → VPC Cloud DB for MySQL

# Task 생성 시나리오

Task Name: classic-to-vpc-db-migration

Source Endpoint: Classic-MySQL-Endpoint

Target Endpoint: VPC-CloudDB-Endpoint

Migration Type: Full load and Change Data Capture(CDC)

# CDC를 통해 원본 DB의 변경 사항이 Target에 실시간으로 반영됩니다.

콘솔 화면을 통해 Source/Target Endpoint를 생성하고, 위 시나리오에 따라 Migration Task를 생성 및 실행합니다.

Session 3.3

수동 데이터 마이그레이션

파일 및 DB 이전 기법

수동 마이그레이션 도구 및 기법

파일/디렉터리 이전

  • rsync: 증분 백업 및 동기화에 최적. SSH 기반으로 안전하게 전송.
  • scp: Secure Copy. 간단한 파일 복사에 용이.
  • Object Storage: 대용량 파일, 비정형 데이터를 임시 저장 후 VPC 서버에서 다운로드.

데이터베이스 이전

  • mysqldump: MySQL/MariaDB의 논리적 백업(덤프) 생성 후 복원.
  • pg_dump: PostgreSQL의 논리적 백업 생성 후 복원.
  • DB 벤더 제공 도구: Oracle Data Pump 등 각 DB가 제공하는 전용 마이그레이션 유틸리티 활용.

예시: rsync와 mysqldump 명령어

rsync를 이용한 웹 소스 동기화

rsync -avz -e "ssh -i mykey.pem" /var/www/html/ root@VPC_IP:/var/www/html/

mysqldump를 이용한 DB 이전

# 1. Classic 서버에서 DB 덤프

mysqldump -u root -p [DB_NAME] > dump.sql

# 2. VPC 서버에서 DB 복원

mysql -u root -p [DB_NAME] < dump.sql

Session 3.4

VPC 네트워크 및 보안

설정 실습

실습: 3-Tier 아키텍처 보안 설정

Internet
VPC
Web Subnet
WAS Subnet
DB Subnet

각 Subnet에 대한 Network ACL과 각 Tier의 서버에 대한 ACG를 설정하여
허용된 트래픽만 통과시키는 계층형 보안을 구축합니다.

Network ACL (Subnet 방화벽) 설정 예시

Subnet Inbound 규칙 Outbound 규칙
Web Subnet Source(0.0.0.0/0), Port(80, 443) Allow 모든 트래픽 Allow
WAS Subnet Source(Web Subnet 대역), Port(8080) Allow Destination(DB Subnet 대역), Port(3306) Allow
DB Subnet Source(WAS Subnet 대역), Port(3306) Allow 모든 트래픽 Deny

ACG (서버 방화벽) 설정 예시

Web Server ACG

Inbound: Source(0.0.0.0/0), Port(80, 443) 허용

WAS Server ACG

Inbound: Source(Web Server ACG), Port(8080) 허용

DB Server ACG

Inbound: Source(WAS Server ACG), Port(3306) 허용

ACG는 Stateful이므로 Outbound 규칙을 별도로 설정할 필요가 없으며, 그룹을 소스로 지정하여 유연하게 관리할 수 있습니다.
